# Plan: Gemini Room Butler Replies When User Is Alone

## Goal
Enable an AI assistant reply flow backed by Gemini in room chat: when exactly one human user is present in a room, Gemini responds to that user after a 5-second delay, always in an extremely polite British butler tone.

## Scope
In scope:
- Server-side Gemini integration for chat replies.
- Room occupancy tracking to detect "single human in room" condition.
- Delayed AI response behavior (5 seconds) before emit.
- Prompt/persona controls to enforce polite British butler style.
- Safe fallback behavior when Gemini is unavailable.
- UI rendering of AI replies using existing message list.
- Test coverage for eligibility logic and error/fallback handling.

Out of scope:
- Persistent bot memory across process restarts.
- Multi-model routing or model A/B testing.
- Content moderation beyond existing validation and error handling rules.
- Billing dashboards or token usage analytics.

## Assumptions
- Gemini API key will be provided via environment variable and not committed.
- Current in-memory storage model remains the source of truth for messages.
- "Only one user in the room" means one connected socket in that room.
- AI messages should appear as normal chat messages with a dedicated nickname (for example, `Reginald`).
- Delay is measured from when a valid user message is accepted by server.
- Existing room names (`general`, `random`, `support`) remain unchanged.

## Open Questions
1. Should Gemini reply to every user message while the room has one human, or only when explicitly mentioned?
Recommended answer: Reply to every valid user message while alone for predictable behavior.
Answer: Reply to every valid user message while the room has exactly one connected user.

2. What nickname should be used for the AI persona?
Recommended answer: `Reginald` to clearly signal a character while staying concise.
Answer: `Reginald`.

3. Should AI responses be capped to a max length?
Recommended answer: Yes, 2-4 short paragraphs or about 500 characters to keep chat readable.
Answer: Not explicitly capped in code for now; rely on prompt instruction for concise output.

4. If a second user joins during the 5-second delay, should the pending AI message still be sent?
Recommended answer: No. Re-check room occupancy right before sending and cancel if user is no longer alone.
Answer: Cancel pending AI message if occupancy changed.

5. Should AI responses be enabled in all rooms or a subset?
Recommended answer: Enable in all rooms for now; keep a room allowlist option in config for later.
Answer: Enabled in all rooms.

Resolved status: Completed in implementation.

## Steps
1. Define AI integration contract
- Add server environment variables in docs and server startup notes:
	- `GEMINI_API_KEY`
	- `GEMINI_MODEL` (default to a stable Gemini chat-capable model)
	- Optional `AI_BUTLER_NAME` and `AI_REPLY_DELAY_MS` (default 5000)
- Create a small server-side AI client module (for example `apps/server/src/ai.ts`) that exposes:
	- `generateButlerReply(input: { room: string; nickname: string; text: string }): Promise<string>`
- Keep provider-specific logic isolated from socket handlers.

2. Add room presence tracking
- Extend socket server logic in `apps/server/src/socket.ts` to track active participants per room.
- Maintain join/leave updates on:
	- `room:join`
	- socket `disconnect`
- Add helper(s) to compute current human count per room and determine AI eligibility.

3. Implement delayed AI reply pipeline
- On `message:send` success path, after storing and broadcasting user message:
	- Check eligibility: exactly one human currently in room.
	- If eligible, schedule delayed task (`setTimeout` for 5000 ms).
	- Re-check eligibility immediately before sending AI reply.
	- Call Gemini client with strict system instruction for tone:
		- Extremely polite language
		- British butler voice
		- Avoid offensive or unsafe content
- Convert model output into standard message object and emit via existing `message:new` flow.
- Store AI message in `chatStore` so history fetch includes bot replies.

4. Add guardrails and failure behavior
- If Gemini request fails, do not crash socket flow.
- Return user-safe behavior:
	- Option A: Silent failure (no AI message)
	- Option B: Fallback canned butler apology message
- Log internal error context server-side without secrets.
- Ensure API key is never emitted to client or logs.

5. Client adjustments (if needed)
- Confirm `apps/client/src/App.tsx` can render AI messages without schema changes.
- Optionally style AI messages with a subtle badge/class only if minimal CSS changes are desired.
- Keep current toast and message rendering behavior intact.

6. Tests
- Add/extend server tests under `apps/server/src/test/` for:
	- Eligibility logic when room has exactly one user.
	- No AI response when room has 2+ users.
	- Delayed send behavior and cancellation when occupancy changes during delay.
	- Gemini client failure path does not break user message delivery.
	- Validation still works for invalid payloads.
- Use deterministic timing controls (fake timers) for delay assertions.

7. Documentation and approval gate
- Update architecture notes if AI assistant becomes part of chat flow boundaries.
- Add short config section describing required env vars and local run expectations.
- Approval received and implementation completed.

## Validation
- Automated checks:
	- `npm run lint`
	- `npm run typecheck`
	- `npm run test`
- Manual scenario checks:
	- Single user in room sends message, receives AI reply after about 5 seconds.
	- Two users in same room: no AI reply to normal user messages.
	- Second user joins during pending delay: pending AI reply is canceled.
	- Gemini outage/misconfig path: app remains usable and user can continue chatting.
	- Message history endpoint includes AI messages generated earlier in process lifetime.
- Non-functional checks:
	- No secrets in repo files, responses, or logs.
	- AI tone remains consistently polite and butler-like across repeated prompts.

## Risks
- Cost and latency risk from external Gemini calls under frequent message traffic.
- Tone drift risk: model may occasionally deviate from butler persona.
- Race conditions around join/leave timing and delayed task execution.
- Duplicate AI replies if retries/timeouts are not carefully bounded.
- Local development friction if API key/model configuration is missing.

## Rollback
- Disable AI replies using config flag or by unsetting `GEMINI_API_KEY`.
- Revert AI reply scheduling path while keeping core human chat flow intact.
- Remove AI client module and related tests if full rollback is needed.
- No database rollback required because storage is in-memory only.
